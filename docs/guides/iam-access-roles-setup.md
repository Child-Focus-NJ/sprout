# IAM Access Roles Setup Guide for Sprout

> The Sprout AWS account currently has only a root user. This guide sets up IAM access using a mix of manual steps (root lockdown) and CDK (everything else).

---

## Step 1: Secure the Root Account (Manual ‚Äî Project Lead)

1. Sign in as root ‚Üí click your name (top right) ‚Üí **Security credentials**
2. Under **MFA** ‚Üí **Assign MFA device** ‚Üí Authenticator app ‚Üí scan QR ‚Üí enter two codes
3. Under **Access keys** ‚Üí delete any that exist
4. Store root email and MFA recovery info in a password manager
5. **Never use root for daily work again**

## Step 2: Set a Password Policy (Manual ‚Äî Project Lead)

**IAM** ‚Üí **Account settings** ‚Üí **Password policy** ‚Üí **Edit**:

- Minimum length: **14** | Require uppercase, lowercase, numbers, symbols: **all yes**
- Password expiration: **90 days** | Prevent reuse: **last 5**

## Step 3: Create the First Admin User (Manual ‚Äî Project Lead)

This is the chicken-and-egg step ‚Äî you need one IAM user before CDK can run.

1. **IAM** ‚Üí **Users** ‚Üí **Create user**
2. Username: the project lead's name
3. Check **Provide user access to the AWS Management Console**
4. Select **I want to create an IAM user**
5. Password: **Autogenerated** ‚Üí check **User must create a new password at next sign-in**
6. Click **Next** ‚Üí skip groups for now (CDK will create them) ‚Üí **Create user**
7. Attach `AdministratorAccess` directly to this user (temporary ‚Äî we'll move them to the CDK-managed group after deploy)
8. Log in as this user ‚Üí set up MFA immediately

## Step 4: Bootstrap CDK (One-Time ‚Äî Admin User)

```bash
cd infra/
pip install -r requirements.txt
cdk bootstrap aws://<ACCOUNT_ID>/us-east-1
```

## Step 5: Deploy the IAM Stack

```bash
cdk deploy SproutIam
```

This creates (from `infra/stacks/iam_stack.py`):

| Resource | What it is |
|----------|-----------|
| `SproutAdmins` group | `AdministratorAccess` + MFA enforcement |
| `SproutDevelopers` group | `PowerUserAccess` + MFA enforcement |
| `SproutViewers` group | `ReadOnlyAccess` + MFA enforcement |
| `SproutRequireMFA` policy | Blocks all actions until user enables MFA |
| GitHub OIDC provider | `token.actions.githubusercontent.com` |
| `SproutGitHubActionsRole` | OIDC role scoped to `Child-Focus-NJ/sprout` repo |

The stack outputs the `GitHubActionsRoleArn` ‚Äî you'll need it for GitHub Actions.

## Step 6: Add Users to Groups

After the stack deploys:

1. **IAM** ‚Üí **User groups** ‚Üí `SproutAdmins` ‚Üí **Add users** ‚Üí add the project lead
2. Remove the `AdministratorAccess` policy that was directly attached in Step 3 (the group policy now covers it)
3. Create IAM users for each team member and add them to the appropriate group

### First login instructions for new users:
1. Sign in at `https://<ACCOUNT_ID>.signin.aws.amazon.com/console`
2. Change your temporary password
3. **Set up MFA** (Security credentials ‚Üí Assign MFA device). You're locked out of everything until you do.
4. Sign out and back in

## Step 7: Configure GitHub Actions

Add this to your CI/CD workflows:

```yaml
permissions:
  id-token: write
  contents: read

steps:
  - uses: aws-actions/configure-aws-credentials@v4
    with:
      role-to-assume: arn:aws:iam::<ACCOUNT_ID>:role/SproutGitHubActionsRole
      aws-region: us-east-1
```

No long-lived access keys. Credentials are temporary session credentials and scoped to your repo.

---

## Ongoing

- **New team member**: Create IAM user ‚Üí add to group ‚Üí share credentials securely
- **Someone leaves**: Delete their IAM user ‚Äî access revoked instantly
- **Role change**: Move user between groups
- **Monthly**: Check **IAM** ‚Üí **Credential report** for users without MFA or stale keys
